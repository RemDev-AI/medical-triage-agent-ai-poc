# IMPORTANT : cette installation vLLM CPU utilise l'index de
# roues dédié (download.pytorch.org/whl/cpu) pour éviter de
# tirer la variante GPU/CUDA par défaut. Version PINÉE
# (reproductible), à la différence des images
# vllm-ci-test-repo (réservées au benchmarking interne vLLM,
# taguées par commit flottant — inadaptées à un déploiement).
#
# Prérequis matériel NON garantis sur un tier HF Space
# "CPU basic" partagé (cf. Dockerfile.hf.vllm-cpu pour la
# commande de diagnostic AVX512/AVX2 à exécuter avant
# d'adopter cette option en production) :
#   - AVX512 recommandé (AVX2 accepté depuis vLLM >= 0.17.0,
#     performances réduites)
#   - gcc/g++ >= 12.3.0 (fourni par Dockerfile)
#
# Si le CPU alloué ne supporte ni AVX512 ni AVX2, vLLM CPU
# backend ne s'installera/fonctionnera pas correctement :
# se rabattre sur l'option A (requirements.txt, model_loader.py).
# ==========================================================

# Index PyTorch CPU : nécessaire pour que torch récupère sa
# variante CPU (pas les paquets nvidia-cuda-*). NB : contrairement
# à une version précédente de ce fichier, on N'UTILISE PAS
# "https://wheels.vllm.ai/0.25.1/cpu" comme --extra-index-url :
# ce chemin n'existe pas pour les releases taguées (seuls
# wheels.vllm.ai/nightly/cpu et wheels.vllm.ai/<commit_hash>/cpu
# sont documentés). De plus, pip (contrairement à uv) fusionne
# --extra-index-url avec l'index par défaut PyPI et ne garantit
# PAS de prioriser la variante CPU — c'est très probablement la
# cause du wheel vLLM "vanilla" installé silencieusement en prod,
# d'où l'AttributeError 'init_cpu_memory_env' observée au runtime.
# Le wheel vLLM CPU est donc épinglé ci-dessous par URL directe
# (méthode officiellement documentée pour les releases taguées),
# ce qui élimine toute ambiguïté de résolution.
--extra-index-url https://download.pytorch.org/whl/cpu

# AI / LLM (inférence via vLLM CPU backend natif)
# ==========================================================
#
# Version pinée et stable, AVEC son suffixe local "+cpu" — ce
# suffixe n'est PAS cosmétique : c'est lui que vLLM inspecte au
# runtime (vllm/platforms/__init__.py, cpu_platform_plugin()) pour
# résoudre current_platform en CpuPlatform. Ce suffixe n'existe
# que sur les wheels réellement compilés avec VLLM_TARGET_DEVICE=cpu
# (cf. vllm/setup.py : `if VLLM_TARGET_DEVICE == "cpu": version +=
# "+cpu"`). Sans lui — ex: si pip installe silencieusement le
# wheel standard "0.25.1" depuis PyPI au lieu du wheel CPU-natif de
# wheels.vllm.ai/0.25.1/cpu, ce qui peut arriver avec deux index en
# présence sans version explicitement suffixée — current_platform
# ne résout AUCUNE plateforme (ni CUDA ni CPU) et
# create_engine_config() lève "RuntimeError: Device string must
# not be empty" au démarrage du moteur (observé en prod).
# Ajuster ce suffixe au besoin selon la dernière version stable
# disponible au moment du déploiement — vérifier
# https://docs.vllm.ai/en/latest/getting_started/installation/cpu/
# pour la version recommandée courante et son suffixe exact.

# vllm==0.17.0
#
# IMPORTANT : on n'utilise PAS "vllm==0.25.1+cpu" tel quel ici,
# car pip a besoin d'un index qui expose EXACTEMENT cette chaîne
# de version, et rien ne garantit qu'il ne préfère pas silencieusement
# une variante "vanilla" trouvée sur PyPI. On épingle donc l'URL
# directe du wheel manylinux CPU publié sur les releases GitHub de
# vLLM (méthode documentée officiellement pour les releases taguées,
# cf. https://docs.vllm.ai/en/latest/getting_started/installation/cpu/) :
# tag cp38-abi3 = compatible toutes versions Python >= 3.8 (dont 3.12,
# cf. image python:3.12-slim du Dockerfile) via l'ABI stable.
# Si vous changez de version vLLM, remplacer l'URL entière ci-dessous
# par celle correspondante trouvée dans les assets de la release
# GitHub (https://github.com/vllm-project/vllm/releases/tag/v0.25.1).
https://github.com/vllm-project/vllm/releases/download/v0.25.1/vllm-0.25.1+cpu-cp38-abi3-manylinux_2_34_x86_64.whl
